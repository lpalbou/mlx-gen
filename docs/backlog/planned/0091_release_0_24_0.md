## Context

Backlog 0086 (Wan UMT5 prompt-embed disk cache + residency + `model_kwargs`
passthrough) and 0087 (`--no-validate-health` + enriched save events) are
implemented and adversarially reviewed (three-cycle review, 2026-07-22) but sit
UNCOMMITTED in the working tree with `pyproject.toml` still at 0.23.1. The
BlackPixel host's warm video worker requests residency through `model_kwargs`
and degrades gracefully on 0.23.1 — the dominant per-scene cost (UMT5 reload,
tens of seconds + ~11 GB transient) stays live until this release ships and the
host pin moves.

## Current code reality

Working tree contains: src/mflux/models/wan/prompt_embed_store.py (new),
wan2_2_ti2v.py encode-path rewrite (numpy tokenization + store + residency),
python_runtime.py model_kwargs (+ cache_key coverage), wan_generate.py /
seedvr2 CLI flags, runtime_events.py save-event enrichment, tests (store unit,
efficiency, router, argparser, python_runtime), CHANGELOG [Unreleased], docs
(api.md, python-integration.md, wan flags in the generate epilog). Full CI
selection green: 1352 passed.

## Problem

The change set rewrites the prompt-encode path of EVERY Wan route (tokenizer
`return_tensors` change + default-ON disk cache) with mock-based tests only —
ADR 0001's enforcement clause requires real-checkpoint smoke evidence before
release claims.

## What we want to do

Cut 0.24.0 through the standard release flow with the smoke evidence recorded.

## Requirements (release checklist, from the cycle-2 review)

1. Real-checkpoint smoke evidence (small frame counts; A14B long runs need
   owner approval per AGENTS.md):
   - (a) cache-miss embeds/output identical to 0.23.1 same-seed baseline;
   - (b) disk-cache-hit run bit-identical to its own miss run;
   - (c) `--no-validate-health` writes a healthy file with the
     `health_check: "skipped"` marker;
   - (d) the opt-in parity test `MFLUX_RUN_LOCAL_WAN_PARITY=1`
     (test_wan_prompt_embeds_match_diffusers_fixture) passes.
2. Version bump pyproject.toml 0.23.1 -> 0.24.0 + `uv.lock` regeneration
   (minor bump: new public API surface — model_kwargs, two Wan constructor
   options, three CLI flags, enriched save events).
3. CHANGELOG: retitle [Unreleased] -> [0.24.0] with evidence citations; keep
   the corrected decode-cost framing (health re-decode ~0.3-0.8 s at
   storyboard size, not seconds).
4. Regenerate llms.txt / llms-full.txt (currently mention none of the new
   flags).
5. Gates: make lint / make check / make test-fast + full CI selection; commit;
   tag via GitHub Actions release workflow (never locally).
6. Post-release: recurrent hygiene — move 0086/0087 (and this item) to
   completed/ with evidence links; notify the BlackPixel pin-bump item
   (blackpixel backlog 0064).

## Candidates to ride this wave (cycle-3 review, 2026-07-22)

- Resolve `None` video params to MODEL-CONFIG defaults inside
  `generate_video`/`generate_outputs` (or export a
  `ModelConfig.resolve_video_defaults()` helper). Today only the CLI's
  `_apply_model_defaults` does this, so the BlackPixel worker had to mirror
  the param->config-key mapping (`apply_wan_video_model_defaults`) — a
  two-repo copy of the mapping table that strains the "defaults in one
  source of truth" rule. Landing it here lets the host delete its copy.
- Export the validated LightX2V sampling recipe (steps 4 / guidance 1.0 /
  guidance_2 1.0 / flow_shift 5.0) next to the profile IDs in
  `lora_validation_registry.py`, so hosts consume instead of transcribing
  (BlackPixel currently hardcodes `WAN_A14B_LIGHTX2V_PROFILE_RECIPE`).
